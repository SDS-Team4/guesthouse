## 작성 기준 요약

- 본 문서는 현재 코드 구현 여부와 무관하게, SRS가 요구하는 기능이 안전하게 운영되기 위해 필요한 보안 설계를 정리한다.
- 기능 분류는 SRS의 사용자 흐름과 역할 분리 기준을 따라 게스트, 호스트, 관리자, 공통 보안으로 재구성했다.
- 세션 기반 인증, guest-web와 ops-web 분리, guest-api와 ops-api 분리, Redis 세션, MySQL, MyBatis, room type 기준 예약과 room 기준 실제 배정이라는 현재 아키텍처 전제를 유지했다.
- 각 행은 실제 취약점 하나를 중심으로 작성했고, 방어 전략은 인증, 인가, 입력검증, 데이터 보호, 세션 보안, DB 보안, API 보안, 파일 보안, 감사, 인프라, 예약 무결성 관점으로 정리했다.
- 코드에서 확인된 endpoint는 실제 경로를 사용했고, 아직 미구현이지만 SRS상 필수 기능은 현재 네이밍 규칙과 초기 API 초안을 따라 `(예상)`으로 표기했다.
- 위험도는 개인정보 노출, 권한 상승, 예약 무결성 붕괴, 운영자 기능 오남용, 감사 불가 상황을 우선적으로 `상`으로 평가했다.
- 예약 관련 항목은 `PENDING`도 재고를 점유한다는 baseline과 `reservation_night` 단위 실제 배정을 전제로 작성했다.
- 전략 근거는 SRS 요구사항 ID, 비즈니스 규칙, 예약 시스템 특성, 현재 기술스택 적합성, 확인된 코드/스키마 구조를 함께 반영했다.

## 구현 근거가 확인된 전략

- 세션 기반 역할 제어는 현재 코드에서 이미 사용 중이다. `shared/shared-auth/.../RoleGuardInterceptor.java`와 `AuthWebMvcConfigurer.java`는 `@RequireRoles`를 기반으로 guest, host, admin 경계를 강제한다.
- 비밀번호 해시 저장과 로그인 실패 추적의 기반 구조도 존재한다. `PasswordEncoderConfig.java`는 Bcrypt를 사용하고, `SessionAuthenticationService.java`와 `UserLoginSecurityMapper.xml`은 실패 횟수와 잠금 메타데이터를 다룬다.
- 세션 쿠키 분리의 베이스라인도 있다. `RedisSessionBaselineConfig.java`는 runtimeName 기반 쿠키명, `HttpOnly`, `SameSite=Lax`를 적용한다.
- 예약 무결성은 현재 구현에서도 핵심 통제다. `guest-api/.../ReservationRequestService.java`와 `ReservationInventoryMapper.xml`은 `FOR UPDATE` 기반 재고 잠금, `PENDING` 상태 점유, `reservation_nights` 생성, 실패 시 롤백 구조를 사용한다.
- host ownership 검증도 구현 근거가 명확하다. `OpsReservationQueryService.java`, `ReservationDecisionService.java`, `ReservationReassignmentService.java`, `OpsRoomBlockCommandService.java`, `OpsPricePolicyCommandService.java`는 host 소유자 또는 admin 여부를 다시 확인한다.
- 감사 로그 구조는 현재 코드와 스키마 모두에 있다. `AuditLogMapper.xml`과 `db/schema-v1.sql`은 actor, target, action, reason, before/after JSON을 저장할 수 있고, 각 `*AuditService`가 이를 활용한다.
- 내부 정보 노출 금지 원칙도 일부 반영돼 있다. `GlobalExceptionHandler.java`는 validation 오류와 예기치 않은 예외를 일반화된 메시지로 반환하고, 스택 트레이스는 서버 로그에만 남긴다.

## SRS 기반으로 설계된 필수 보안 전략

- 계정 복구는 아직 부분 미구현이어도 SRS상 필수다. 아이디 찾기와 비밀번호 재설정에는 해시 저장된 단회성 인증코드, TTL, 재전송 제한, 시도 횟수 제한, 검증 완료 후 단기 세션 권한 모델이 필요하다.
- 세션 기반 인증을 채택한 이상 CSRF 방어는 필수다. 예약 요청, 취소, 운영 변경, 관리자 기능에는 CSRF 토큰과 `Origin` 검증을 함께 적용해야 한다.
- 관리자 영역은 일반 사용자 영역보다 강한 제어가 필요하다. `REQ-SEC-007`에 따라 운영 환경에서는 IP allowlist 또는 VPN 경계를 두고, 중요 기능에는 재인증을 붙이는 것이 타당하다.
- 파일 업로드가 도입되면 UUID 기반 저장명, 확장자와 MIME 및 magic number 검증, 간접 다운로드, 오브젝트 스토리지 메타데이터 분리 모델을 적용해야 한다.
- 공지와 INFO 텍스트는 저장형 XSS 방어가 핵심이다. HTML 비허용 또는 sanitizer, 출력 인코딩, CSP를 함께 적용해야 한다.
- 감사 로그는 조회 기능만 있는 수준을 넘어서 append-only 성격을 가져야 한다. 운영 API에서 삭제나 수정이 가능하면 감사 자체가 무력화된다.
- 예약 관련 write API에는 idempotency key와 rate limit를 붙여야 한다. 이는 보안과 성능을 동시에 지키는 현실적인 제어다.
- 관리자 회원 조회와 시스템 로그 조회에는 데이터 최소 공개 원칙이 필요하다. 전체 조회 권한과 내부 비밀값 조회 권한은 구분해야 한다.

## 추가 보완 권장 사항

- 로그인 실패 정책은 현재 실패 횟수 추적까지는 보이지만, 임계치 초과 시 `locked_until`을 설정하는 강제 로직은 더 분명히 구현할 필요가 있다.
- 세션 보안은 현재 `HttpOnly`와 `SameSite`까지는 보강 근거가 있으나, 운영 HTTPS 환경에서 `Secure` 속성과 로그인 직후 세션 ID 회전까지 반드시 포함하는 편이 안전하다.
- 현재 구조는 소유권 검증과 감사 로그가 잘 잡혀 있지만, 계정 복구, 공지/약관, 첨부 업로드, 관리자 감사로그 조회 API는 SRS상 요구 대비 아직 설계 선행 항목이 남아 있다.
- admin IP 제한은 baseline 문서에서 후속 마일스톤으로 미뤄져 있지만, 평가 기준상 관리자 영역의 보안 강화를 문서와 배포 체크리스트에 명시적으로 포함하는 것이 좋다.
- 예약 조회와 운영 로그 조회에는 trace ID 기반 상관관계 추적을 붙여 장애와 오남용 분석 시간을 줄이는 것이 바람직하다.
- guest와 ops 프런트에서 원시 HTML 렌더링을 금지하고, 특히 공지/INFO/운영 메모 계열 텍스트는 공통 sanitize 정책을 두는 것을 권장한다.
