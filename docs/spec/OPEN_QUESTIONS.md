## OQ-01 동시성 전략 명확화
SRS 일부 구간에는 "낙관적 Lock" 표현이 있으나, 다른 구간과 시퀀스/요구사항은 DB 락과 트랜잭션 기반 처리를 강하게 전제한다.

### 기본 제안
- 예약 요청 시점에 DB 트랜잭션 + 행 단위 락 기반으로 처리
- 구현 패턴은 MySQL 기준 `SELECT ... FOR UPDATE` 또는 이에 준하는 비관적 락을 기본안으로 검토
- "optimistic"이라는 표현은 문서 상 용어 혼선으로 보고 정정 여부 확인

## OQ-03 예약 대기 상태 유지 시간 제한
대기(PENDING) 상태 예약이 언제까지 재고를 점유하는지 정책이 없다.

### 결정 필요
- 무기한 점유인지
- 일정 시간 이후 자동 해제인지
- 호스트 수동 처리 전까지 유지인지

## OQ-04 관리자/호스트 권한 차등화 수준
호스트와 관리자의 권한 차이가 서술상 존재하지만, 실제 메뉴와 API 단위 세부 범위가 더 필요하다.

### 결정 필요
- 관리자도 host UI를 읽기 전용으로 재사용할지
- 별도 admin-only 엔드포인트를 어디까지 둘지

## OQ-06 이미지 업로드/파일 저장 방식
숙소 대표 이미지 및 파일 저장소 방식이 확정되지 않았다.

### 결정 필요
- 로컬 파일 시스템
- 오브젝트 스토리지
- 메타데이터 테이블 구조

## OQ-07 가격 정책 충돌 규칙
중복 기간 정책이 있을 때 우선순위가 불명확하다.

### 결정 필요
- 최신 정책 우선
- 우선순위 숫자 컬럼 추가
- 중복 등록 자체 금지

## OQ-08 게스트 취소 정책의 정확한 기준 시각
SRS에는 "체크인 전까지" 표현이 있으나 정확한 cutoff가 없다.

### 결정 필요
- 체크인 시각 직전
- 체크인일 00:00 이전
- 호스트 숙소별 체크인 시간 기준

## OQ-09 실제 객실 배정 초기값
예약 생성 시 초기 실제 배정을 생성한다고 되어 있으나, 어떤 규칙으로 room을 선택하는지 명확하지 않다.

### 기본 제안
- 단순 first-fit
- 변경 가능성을 전제로 최소한의 deterministic rule 사용
- 이후 호스트가 reservation_night별 수정 가능

## OQ-10 공지 / 약관 / 알림의 초반 범위
SQL 초안엔 NOTIFICATION, TERM, ATTACHMENT가 있으나, 사용자-facing 알림 정책은 약하다.

### 결정 필요
- V1 필수 범위는 공지/약관 관리까지만
- 사용자 알림(Notification)은 후속 범위로 미룰지
