# SPEC_BASELINE

이 문서는 SRS를 Codex가 빠르게 이해하도록 만든 요약본이다.  
세부 구현 시에는 반드시 원문 SRS를 다시 확인한다.

## 1. 제품 목표

게스트가 숙소를 검색하고 예약할 수 있고,  
호스트가 숙소/객실/예약을 운영하며,  
관리자가 전체 회원/권한/로그/공지/약관을 관리하는 웹 기반 시스템을 만든다.

## 2. 핵심 사용자

- 게스트
- 호스트
- 시스템 관리자

## 3. 1차 릴리즈 범위

### 포함
- 게스트 회원가입 / 로그인 / 로그아웃 / 마이페이지
- 호스트 권한 요청
- 아이디 찾기 / 비밀번호 찾기 / 비밀번호 재설정
- 숙소 검색 / 상세 조회 / 객실 타입별 가능 여부 / 예약 현황
- 객실 타입 기준 예약 요청
- 예약 대기 / 취소 / 조회
- 호스트 로그인 / 계정 관리
- 숙소 / 객실 타입 / 개별 객실 / 가격 정책 / block 관리
- 호스트 예약 확정 / 취소 / 실제 객실 배정 변경
- 관리자 회원 관리 / 권한 요청 처리 / 감사 로그 / 시스템 로그 / 운영 현황 / 약관 / 공지
- 감사 로그와 운영 이력

### 제외
- PG 결제 연동
- OTA 양방향 재고 연동
- 모바일 앱
- 다국어
- 자동 객실 최적 배정 알고리즘

## 4. 중요한 도메인 규칙

### 예약
- 예약은 객실 타입 기준으로 접수된다.
- 실제 운영 배정은 개별 객실 기준이다.
- 연박 중 날짜별로 서로 다른 객실 배정이 가능하다.
- 예약 요청 시 재고가 즉시 점유된다.
- 검색/상세 조회 단계에서는 재고를 점유하지 않는다.
- 대기 상태도 재고 점유로 간주한다.
- 초과 예약은 허용되지 않는다.

### 권한
- 게스트는 자기 예약/정보만 접근 가능
- 호스트는 자기 숙소/객실/예약만 접근 가능
- 관리자는 전체 접근 가능
- 게스트는 직접 객실 변경 불가
- 호스트만 최종 확정/취소/배정 변경 가능

### 삭제/비활성화
- 운영 이력이 있는 자산은 삭제보다 비활성화 우선

## 5. 아키텍처 기준

- guest 서비스와 ops(host/admin) 서비스는 분리 운영
- Spring Boot + MyBatis + MySQL + React 방향
- 세션 기반 인증
- 비밀번호 Bcrypt
- JWT 제외
- Lombok 제외
- Nginx 리버스 프록시 / 분리 배포 전제

## 6. 성능/보안 기준

### 성능
- 숙소 목록 조회: 일반 조건 3초 이내
- 숙소 상세 조회: 일반 조건 2초 이내
- 예약 요청의 재고 점유 반영: 3초 이내
- 관리자 예약 목록 조회: 3초 이내

### 보안
- 로그인 실패 보호 정책 필요
- HTTPS 필요
- 세션 안전 생성/종료 필요
- 내부 정보 노출 금지
- 객체 단위 접근통제 필요

## 7. 구현 우선순위 추천
