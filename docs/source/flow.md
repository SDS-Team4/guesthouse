# flow.md

# Guesthouse Reservation System - Codex Friendly Flow Spec

# Source: SRS 3.x + 4.x + Appendix B.4/B.7 통합본

## 0. 문서 목적

이 문서는 SRS의 UI 정의(3장), 기능 요구사항(4장), 분석 모델의 Sequence / DFD를
Codex가 구현 기준으로 바로 사용할 수 있게 재구성한 흐름 문서다.

이 문서의 목적은 다음과 같다.

1. 화면 이동 흐름을 번호 기준으로 고정한다.
2. 사용자 역할별(Guest / Host / Admin) 행동 범위를 고정한다.
3. 예약 처리 시점의 상태 전이와 데이터 처리 순서를 고정한다.
4. 현재 코드베이스와 SRS 간 상충 여부를 검토할 기준 문서로 사용한다.
5. 공용 CSS / 공용 컴포넌트 제작 시 기준 화면 목록으로 사용한다.

## 1.0 게스트 인증 및 회원 관리

### 1.1 회원가입

- 화면: `UI-GUEST-001`
- 목적: 비회원이 게스트 계정을 생성한다.
- 입력:
  - loginId
  - password
  - passwordConfirm
  - name
  - email
  - phone
  - terms agreement
- 처리:
  1. 아이디/이메일 중복 검증
  2. 비밀번호 정책 검증

  4. Guest 계정 생성
- 결과:
  - 성공 시 `UI-GUEST-002` 로그인 화면으로 이동
  - 실패 시 현재 화면 검증 메시지 표시

### 1.2 로그인 / 세션 생성

- 화면: `UI-GUEST-002`
- 목적: 게스트 로그인 및 세션 생성

  1. 계정 조회
  2. 비밀번호 검증
  3. 세션 생성
- 결과:
  - 성공 시 메인 검색 화면 또는 직전 보호 화면 복귀
  - 실패 시 오류 메시지 표시

### 1.3 로그아웃

- 처리:

  2. `UI-GUEST-003`으로 이동

## 3.0 예약 요청, 점유, 조회 및 취소

### 3.1 예약 진입

- 화면: `UI-GUEST-006`
- 진입 조건:
  - 로그인된 Guest
  - 예약 가능한 객실 타입 선택 완료
- 비로그인 시:
  - `UI-GUEST-002` 로그인 화면으로 리다이렉트 후 복귀

### 3.2 예약 요청 처리

- 화면: `UI-GUEST-006`
- 핵심 규칙:
  1. 예약은 객실 타입 기준으로 생성한다.
  2. 예약 버튼 클릭 시에만 재고를 점유한다.
  3. 전체 처리는 하나의 DB transaction으로 수행한다.
  4. 재고 검증 단계부터 lock을 획득한다.
  5. 처리 순서는 다음과 같다.
     - 재고 재검증
     - reservation 생성
     - reservation_nights 생성
  6. 성공 시 상태는 `PENDING`
  7. 실패 시 rollback
- 데이터 구조:
  - reservation: 예약 원본 정보
  - reservation_nights: 숙박일별 실제 배정 정보

### 3.3 예약 완료

- 화면: `UI-GUEST-007`
- 표시:
  - 예약 번호
  - 숙소명
  - 객실 타입명
  - 날짜
  - 상태(PENDING)

### 3.4 예약 목록 조회

- 화면: `UI-GUEST-010`
- 처리:
  1. 로그인 Guest 본인 예약만 조회
  2. 상태/기간 필터 가능

### 3.5 예약 상세 조회

- 화면: `UI-GUEST-011`
- 표시:
  - 숙소 정보
  - 객실 타입
  - 날짜
  - 인원수
  - 예약 상태
- 금지:
  - 실제 객실 번호 기본 노출 금지
  - 내부 운영 배정 결과 기본 노출 금지

### 3.6 예약 취소

- 화면: `UI-GUEST-011`
- 처리:
  1. 본인 예약 여부 확인
  2. 취소 가능 상태 검증
  3. reservation 상태를 `CANCELLED`로 변경
  4. 관련 점유/배정 상태 갱신
- 결과:
  - 성공 시 취소 완료 메시지 표시

### 6.3 예약 확정

- 조건:
  - Host 소유 숙소 예약
  - 확정 가능 상태
- 처리:
  1. reservation 상태 변경
  2. 감사 로그 기록

- 조건:
  - Host 소유 숙소 예약

  2. 필요 시 재고 복구
  3. 감사 로그 기록

### 6.5 실제 객실 배정 변경

- 화면: `UI-HOST-009`
- 핵심 규칙:
  1. Guest는 불가, Host만 가능
  2. reservation_nights 단위로 room_id 변경
  3. 자동 최적화가 아니라 수동 조정 우선
  4. 동일 예약의 연속된 일자는 하나의 연박 블록처럼 보여야 한다.
  5. block 영역 또는 이미 점유된 칸에는 드롭할 수 없다.
  6. 확정 가능 여부는 연속 배정 성립 상태를 반영할 수 있어야 한다.
  7. 변경 전/후 room, 시간, 사유를 이력으로 저장한다.

### 6.6 재고 그리드

- 화면: `UI-HOST-009`
- 구성:
  - 객실 타입별 아코디언

  - 가로축 = date
  - 기본 1주 노출

## 8.0 공통 구현 규칙

### 8.1 아키텍처

- Guest 서비스와 Host/Admin 서비스는 서로 다른 진입점으로 제공한다.
- Guest 서비스와 Host/Admin 서비스는 논리적으로 분리된 runtime 기준으로 본다.

### 8.2 인증

- 세션 기반 인증 사용
- JWT 사용하지 않음
- 비밀번호는 bcrypt 해시 저장

### 8.3 권한

- Guest:
  - 본인 계정 / 본인 예약만 접근 가능
- Host:
  - 본인 소유 숙소/객실/예약만 접근 가능
- Admin:
  - 전체 회원 / 전체 숙소 / 전체 예약 접근 가능

### 8.4 예약 규칙

- 예약은 객실 타입 기준으로 접수
- 실제 객실 배정은 reservation_nights 기준
- Guest는 실제 객실 변경 불가
- Host는 실제 객실 변경 가능

### 8.5 재고 / 동시성

- 검색/상세 조회 단계에서는 점유 금지
- 예약 버튼 클릭 시점부터 점유
- lock + transaction + rollback 규칙 준수

### 8.6 삭제 정책

- 예약 이력 존재 시 즉시 삭제보다 비활성화 우선

### 8.7 감사 로그

- Host 주요 행위는 모두 감사 로그 대상
- actor / target / timestamp / reason-or-diff 저장

## 9.0 Codex 작업 규칙

1. 현재 코드베이스의 route / page / component / API / enum / DB schema를 이 문서 기준으로 검토한다.
2. 각 구현 항목은 반드시 하나 이상의 UI ID 또는 요구사항 번호와 연결한다.
3. SRS와 상충하는 동작은 flow.md 쪽을 기준으로 수정 후보를 만들고, 변경 이유를 문서화한다.
4. 새 기능을 추가하지 말고, 먼저 기존 기능을 이 흐름에 맞춰 정렬한다.
5. Guest / Host / Admin 공용 CSS 작업은 반드시 이 문서의 UI ID 기준으로 매핑한다.
